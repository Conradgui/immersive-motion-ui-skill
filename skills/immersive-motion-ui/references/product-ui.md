# Product UI

Load this for dashboards, admin tools, analytics pages, SaaS apps, settings, tables, filters, app shells, and repeated operational workflows.

## Job

Product UI is for repeated work. It should be calm, dense, stable, and fast to scan. Premium product UI is not cinematic; it is precise, legible, and hard to break.

## Layout Bias

Prefer:

- persistent navigation
- clear page title and state
- dense but readable tables, charts, lists, and forms
- visible filters and sorting
- stable dimensions for controls, panels, cards, and data cells
- compact hierarchy instead of oversized hero sections

Avoid:

- marketing hero layout inside an app
- decorative cards inside other cards
- huge empty panels with low information value
- motion that delays repeated actions
- hiding controls behind clever interactions

## Density Rules

Increase density by improving structure, not by shrinking text until it breaks.

Use:

- predictable grid tracks
- fixed control heights
- tabular numbers where useful
- grouped filters
- clear empty, loading, error, and selected states
- restrained color for status and priority

Do not:

- scale font size with viewport width
- rely on hover-only controls for core actions
- let labels wrap unpredictably in fixed controls
- use decorative gradients as status encoding

## Product Polish

High-quality product UI usually shows:

- named primary workflow
- obvious current state
- meaningful default view
- clear action hierarchy
- accessible focus and keyboard paths
- responsive behavior that preserves task order
- no layout shift when counts, labels, filters, or loading states change

## Motion

Use motion for:

- panel open/close continuity
- row expansion
- filter result transitions
- drag/drop or reordering feedback
- loading skeletons with reduced-motion support

Avoid:

- cinematic page entrances on every route
- scroll hijacking
- large parallax effects
- animated backgrounds behind dense information

## Review Questions

Ask:

1. Can a returning user complete the main task faster?
2. Are controls where an experienced user expects them?
3. Does the layout remain stable under long labels and real data?
4. Is status color readable without relying only on color?
5. Did we add polish without reducing information clarity?
