# Showcase Casebook

Load this after `industry-directions.md` when a real comparison would improve design judgment. Match the task and failure mechanism here, then load only the selected case. Transfer the reasoning rather than the visual answer.

## Comparison Protocol

1. Match by job-to-be-done and failure mechanism, not by industry label alone.
2. Keep the brief, content, data, stack, and constraints fixed across the pair.
3. Identify what the before gets plausibly right before naming the harm. Before 不是稻草人。
4. Trace each improvement through problem, intervention, and user outcome.
5. Treat the after as one defensible resolution. After 不是唯一正确答案。
6. State evidence that would justify another direction.
7. Keep browser claims at `NOT EXECUTED` until a runnable surface is actually inspected.

风格不是质量代理。Dark, minimal, editorial, glass, 3D, or high-motion treatments may be appropriate when the subject and task support them. 用户建议是方向证据，不是永久限制；respect the intent, but do not turn one example into a permanent constraint.

## Transfer Gate

Before using any case, answer:

1. Is the task job and failure mechanism actually similar, or only the industry label?
2. Which fixture facts differ from the user's real project?
3. Which causal delta transfers, and which one does not?
4. What evidence would make the rejected direction appropriate?
5. Are you borrowing a reasoning move or imitating the case's surface?
6. Which browser, responsive, motion, accessibility, and performance checks remain `NOT EXECUTED`?

If the answer is mostly visual resemblance, do not use the case. Return to the real brief and project evidence.
